# Phase 04 Plan 04.2: CBCL Database Schema Summary

## One-Liner
Complete CBCL database schema implementation with cbcl_assess table, report_record migration support, and full CRUD API.

## What Was Built

### Task 1: CBCL Assessment Table Schema
**File:** `src/database/init.ts`

Added `cbcl_assess` table with comprehensive field structure:
- **Core fields:** `id`, `student_id`, `age_months`, `gender` (CHECK: male/female)
- **Social Competence (Part 1):** `social_competence_data` (JSON TEXT), `social_activity_score`, `social_social_score`, `social_school_score`
- **Behavior Problems (Part 3):** `raw_answers` (JSON TEXT for 113+8 question answers)
- **Calculated Scores:** `behavior_raw_scores` (JSON), `factor_t_scores` (JSON)
- **Summary Scores:** `total_problems_score`, `total_problems_t_score`, `internalizing_t_score`, `externalizing_t_score`
- **Classification:** `summary_level` (CHECK: normal/borderline/clinical)
- **Metadata:** `start_time`, `end_time`, `created_at`
- **Indexes:** `idx_cbcl_assess_student`, `idx_cbcl_assess_created`
- **Also added:** Missing SRS-2 indexes for consistency

### Task 2: Report Record Migration
**File:** `src/database/migrate-report-constraints.ts`, `src/database/init.ts`

Updated CHECK constraint to include 'cbcl' report type:
- Migration script now handles 'cbcl' type in table recreation
- `needsMigration()` detects missing 'cbcl' in existing constraints
- Log messages updated to show all supported types
- Fresh database schema includes 'cbcl' in report_record CHECK constraint

### Task 3: CBCL Database API
**File:** `src/database/api.ts`

Created `CBCLAssessmentAPI` class extending `DatabaseAPI`:
- `createAssessment(data)` - Insert new assessment with all fields
- `getAssessment(id)` - Get by ID with automatic JSON field parsing
- `getStudentAssessments(studentId)` - List by student (summary view)
- `getLatestAssessment(studentId)` - Get most recent assessment
- `updateAssessment(id, data)` - Partial update support
- `deleteAssessment(id)` - Delete by ID

JSON fields automatically parsed in get methods:
- `social_competence_data`
- `raw_answers`
- `behavior_raw_scores`
- `factor_t_scores`

## Key Technical Decisions

1. **JSON Field Strategy:** Used TEXT columns with JSON storage for dynamic factor scores and raw answers, matching the pattern used in srs2_assess and other assessment tables.

2. **CHECK Constraints:** Added at database level for `gender` (male/female) and `summary_level` (normal/borderline/clinical) to ensure data integrity.

3. **Index Strategy:** Created student_id index for foreign key lookups and created_at DESC index for recent-first listing.

4. **API Design:** Followed existing patterns from ConnersPSQAPI and SRS-2 implementation, with automatic JSON parsing for convenience.
