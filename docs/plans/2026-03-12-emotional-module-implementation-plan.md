# 情绪行为调节模块实施计划

## 1. 目标

在当前 SCGP 系统中，将已预留的 `emotional` 模块从实验状态推进为可交付训练模块，首期完成：

1. 模块入口接入
2. 情绪与场景训练流程
3. 表达关心训练流程
4. 本地记录落库
5. 情绪模块专属记录与报告页面

## 1.1 Roadmap 对齐

本专题已对齐到当前 `.planning/ROADMAP.md` 中的实际阶段编号：

- `Phase 2.2: Emotional Behavior Regulation Module`

原因：

1. 当前 ROADMAP 中已存在 `Phase 2` 与 `Phase 2.1`
2. 情绪行为调节模块属于 Emotional / Social 扩展主线
3. 当前最合适的插入位置是 `2.1` 与 `3` 之间

因此后续所有 GSD 命令、计划文件和验证动作，默认都应使用 `2.2`，不再使用之前文档中的示例编号。

## 2. 实施原则

1. 优先复用现有模块入口、学生选择、训练记录、报告与资源体系。
2. 首期先完成最小闭环，不被素材总量拖住。
3. 数据结构优先保证后续统计能力。
4. 先通路，再打磨内容，再做可视化增强。

## 3. 阶段拆分

## 阶段一：模块激活与入口打通

目标：

- 让情绪模块可以在当前系统中被正常进入

工作项：

1. 调整 `ModuleRegistry` 中 `emotional` 模块定义
2. 确定模块入口是否继续复用 `/games/menu`
3. 增加情绪模块首页和基础路由
4. 打通从模块卡片到学生选择页再到训练页的跳转

产出：

- 情绪模块菜单可见
- 情绪模块基础页面骨架可访问

验收：

- 教师可从现有训练入口进入情绪模块
- 路由跳转完整，不出现死链

## 阶段二：资源模型与演示内容接入

目标：

- 建立可扩展的场景内容结构，并导入首批演示数据

工作项：

1. 定义 `emotion_scene` / `care_scene` 资源类型
2. 约定 `meta_data` JSON 结构
3. 设计首批演示场景内容
4. 编写种子数据或导入脚本
5. 打通资源查询 API

产出：

- 首批情绪场景资源
- 首批表达关心资源
- 资源查询接口

验收：

- 前端可按模块和资源类型正确加载场景
- 元数据结构可扩展到 80+/60+ 场景

## 阶段三：情绪与场景训练流程

目标：

- 完成首个子模块的端到端训练闭环

工作项：

1. 场景展示页实现
2. 情绪五选一交互实现
3. 即时反馈与提示机制实现
4. 社交推理题流程实现
5. 应对方案选择与解释实现
6. 单次会话结果汇总

产出：

- `EmotionSceneTraining.vue`
- 会话总结逻辑

验收：

- 学生可完成完整训练
- 正确、错误、提示、重试逻辑全部可运行

## 阶段四：表达关心训练流程

目标：

- 完成第二个子模块的端到端训练闭环

工作项：

1. 情境展示实现
2. 三类话术选择实现
3. 使用效果说明实现
4. 表达者视角流程实现
5. 接收者视角流程实现
6. 会话汇总与继续训练入口

产出：

- `CareExpressionTraining.vue`

验收：

- 用户可完成两种视角的训练
- 系统可记录话术选择和偏好

## 阶段五：记录落库与查询接口

目标：

- 将训练过程数据结构化保存并可查询

工作项：

1. 扩展 `training_records` 写入情绪模块会话数据
2. 新增 `emotional_training_session`
3. 新增 `emotional_training_detail`
4. 编写保存与查询 API
5. 增加必要的迁移脚本

产出：

- 数据表
- API
- 基础统计查询

验收：

- 训练后数据可重启后读取
- 可按学生、子模块、日期查询记录

## 阶段六：记录页与报告页

目标：

- 让教师和家长能查看训练历史和趋势

工作项：

1. 实现情绪模块记录页
2. 实现情绪模块专属报告页
3. 接入 ECharts 图表
4. 将摘要接入总报告入口
5. 生成老师/家长可读建议

产出：

- `Records.vue`
- `Report.vue`

验收：

- 至少输出 5 类图表
- 报告数据与明细一致

## 阶段七：体验打磨与验证

目标：

- 保证特殊教育交互要求真正落地

工作项：

1. 单屏单任务检查
2. 按钮和触屏尺寸检查
3. 阶梯式提示文案检查
4. 正向反馈文案检查
5. 中断续做能力检查
6. 类型检查与回归测试

产出：

- 可交付版本
- 验证记录

验收：

- `npm run type-check` 通过
- 主要训练路径可手工验证

## 4. 推荐开发顺序

推荐按以下顺序推进：

1. 入口和路由
2. 资源结构
3. 情绪与场景
4. 表达关心
5. 数据落库
6. 记录和报告
7. UI 与特教交互打磨

原因：

- 先打通可访问路径，避免后面页面无处挂接
- 先确定资源模型，避免 UI 结构反复返工
- 先实现训练闭环，再做统计和报告

## 5. 首期 MVP 范围

建议 MVP 只要求：

1. 情绪模块从 `experimental` 变为可访问
2. 每个子模块各提供 6-10 个高质量演示场景
3. 完成完整作答、反馈、提示和保存
4. 完成最基础记录页
5. 完成最基础报告页

暂缓项：

1. 教师后台内容编辑器
2. 批量资源导入工具
3. 复杂导出模板
4. 高级个性化推荐

## 6. 风险与应对

### 风险一：素材数量先行导致开发阻塞

应对：

- 先做扩展型数据结构
- 首期只放少量高质量演示数据

### 风险二：全部塞进 `raw_data` 导致后续报表困难

应对：

- 首期就引入结构化明细表

### 风险三：页面过于复杂，不符合特殊教育交互

应对：

- 每个训练页只保留一个主任务区
- 使用分步状态机组织流程

### 风险四：硬编码内容过多，后续维护困难

应对：

- 所有场景题干、选项、话术都走资源数据

## 7. 交付清单

首期建议交付：

1. 产品文档
2. 页面与路由设计
3. 数据设计
4. 数据迁移与 API
5. 前端训练页面
6. 记录页
7. 报告页
8. 演示数据
9. 验证记录

## 8. 完成定义

满足以下条件视为首期完成：

1. 教师可从现有系统入口启动情绪模块训练
2. 两个子模块都能完整训练并保存结果
3. 数据可在记录页和报告页被查看
4. 至少具备基础趋势图和偏好图
5. 页面符合特殊教育低负荷交互要求
