# 2026-03-13 SCGP 可执行路线图

> **文档类型**: 现行路线图
> **基线来源**:
> - `docs/planning/2026-03-13-scgp-current-prd.md`
> - `docs/reports/2026-03-13-scgp-prd-gap-analysis.md`
> **目标**: 把当前 PRD 差距拆成可执行、可验收、可排期的阶段工作

## 1. 路线图原则

本路线图按以下原则制定：

- 先收口平台底座，再放大未来模块
- 先解决数据可靠性问题，再解决体验一致性问题
- 先让“现有主链更稳”，再让“平台形态更完整”
- 不把未来模块扩张放在底座缺口之前

## 2. 里程碑总览

### Milestone A: 平台底座收口

目标：

- 让备份、资源文件、收藏等基础能力与当前主线 schema 和资源模型真正一致

包含阶段：

- Phase A1 备份与恢复收口
- Phase A2 资源文件生命周期收口
- Phase A3 资源收藏与资源中心体验收口

### Milestone B: 评估与资源中心平台化

目标：

- 让“统一评估架构”和“统一资源中心”从技术抽象进一步落成产品层统一行为

包含阶段：

- Phase B1 评估入口配置化
- Phase B2 报告与评估页面公共骨架收口
- Phase B3 资源中心产品化收口

### Milestone C: 模块装配平台化

目标：

- 让模块注册表真正开始驱动平台导航、模块开放状态和后续模块扩展

包含阶段：

- Phase C1 模块首页与菜单注册表驱动
- Phase C2 路由与模块开放机制收口
- Phase C3 开发/迁移工具隔离与命名清理

## 3. 详细阶段

## Phase A1: 备份与恢复收口

### 目标

让备份/恢复能力覆盖当前主线实际在用的数据结构，而不是停留在旧表阶段。

### 范围

- 梳理当前备份导出覆盖表
- 补齐新资源表覆盖
- 补齐班级相关表覆盖
- 补齐报告索引和当前主线业务记录覆盖
- 校验恢复事务和失败回滚

### 交付物

- 更新后的备份/恢复实现
- 备份覆盖清单文档
- 恢复验证说明

### 验收标准

- 备份后恢复，资源中心数据不丢
- 备份后恢复，班级与分班数据不丢
- 备份后恢复，报告索引和训练记录可用
- 恢复失败时不会破坏当前库

### 风险

- 旧表与新表并存可能导致重复导出或漏导出
- 若恢复策略不清晰，可能造成同名表覆盖错误

## Phase A2: 资源文件生命周期收口

### 目标

把资源元数据与物理文件操作收口到统一主链，减少孤儿文件和引用失效。

### 范围

- 明确统一的资源文件服务
- 收口资源新增、替换、删除、恢复、硬删除
- 明确资源文件目录约定
- 校验 `resource://` 引用与物理文件一致性

### 交付物

- 统一资源文件服务层
- 文件保存/删除主链收口
- 生命周期说明文档

### 验收标准

- 新增资源后可立即访问
- 删除资源后不会留下不可追踪引用
- 硬删除资源后无残留无效记录
- 恢复资源后封面和文件路径可再次访问

### 风险

- 迁移期需要兼容旧路径
- 需要避免把物理文件误删到预置资源目录

## Phase A3: 资源收藏与资源中心体验收口

### 目标

完成资源收藏能力，并让资源中心在用户体验层闭环。

### 范围

- 完成收藏查询能力
- 完成收藏入口与筛选
- 明确管理员与教师在资源中心的差异行为
- 优化资源中心入口文案和筛选逻辑

### 交付物

- 收藏能力完整主链
- 资源中心交互收口
- 角色差异说明

### 验收标准

- 用户可收藏/取消收藏资源
- 可筛选“仅收藏”
- 管理员和教师的交互行为一致且可解释

## Phase B1: 评估入口配置化

### 目标

把评估中心从“硬编码量表卡片”推进到“可配置入口”。

### 范围

- 评估选择页配置化
- 量表显示信息集中管理
- 入口状态与可用性统一判断

### 交付物

- 配置化评估入口
- 量表元数据集中定义

### 验收标准

- 新增量表入口时不再需要直接修改大段模板结构
- 已接入量表的入口信息统一来源
- 不可用量表能明确显示状态

## Phase B2: 报告与评估页面公共骨架收口

### 目标

继续减少量表间重复代码，让统一评估架构在报告层也开始体现。

### 范围

- 抽公共报告骨架
- 收口公共导出与公共页头信息
- 统一报告页基础字段

### 交付物

- 公共报告骨架组件或协议
- 报告页统一字段约束

### 验收标准

- 多量表报告页共享统一基础布局
- 报告页公共字段定义不再分散漂移

## Phase B3: 资源中心产品化收口

### 目标

让资源中心真正成为统一入口，而不是“统一查询 + 若干历史兼容”。

### 范围

- 收敛旧资源入口
- 完善训练资源/教学资料分区模型
- 优化空态、筛选态、状态提示

### 交付物

- 更稳定的资源中心信息架构
- 历史入口清理计划

### 验收标准

- 用户能明确区分资源类型和用途
- 历史入口不再与当前入口并行造成歧义

## Phase C1: 模块首页与菜单注册表驱动

### 目标

让模块元数据开始真正驱动平台首页或模块首页。

### 范围

- 模块卡片由注册表生成
- 模块状态统一展示
- 对 `active`、`experimental`、`deprecated` 有明确 UI 语义

### 交付物

- 注册表驱动的模块菜单或模块首页

### 验收标准

- 菜单和模块状态不再双重维护
- 未来模块能以统一方式暴露“开发中”状态

## Phase C2: 路由与模块开放机制收口

### 目标

把“模块是否开放”和“用户能否进入”变成统一机制，而不是散落在页面和路由里。

### 范围

- 模块开放状态判断收口
- 模块状态与路由守卫联动
- 未来模块占位页统一化

### 交付物

- 模块开放机制
- 模块状态路由守卫

### 验收标准

- 用户点击实验模块时行为一致
- 平台对外能明确表达模块开放状态

## Phase C3: 开发工具隔离与命名清理

### 目标

把调试/迁移工具与正式产品边界分开，并完成系统命名残留清理。

### 范围

- 收敛 devtools 路由
- 清理旧系统名默认值
- 清理缓存 key 和元数据中的旧命名

### 交付物

- 干净的主产品入口
- 命名统一清单

### 验收标准

- 正式产品路径不再混入大量测试页
- 用户可见文案统一使用 `SCGP`
- 保留必要兼容，但不再对外暴露旧命名

## 4. 推荐执行顺序

推荐按以下顺序推进：

1. Phase A1
2. Phase A2
3. Phase A3
4. Phase B1
5. Phase B2
6. Phase B3
7. Phase C1
8. Phase C2
9. Phase C3

理由：

- A 阶段优先解决数据可靠性与基础资源能力
- B 阶段优先解决用户可感知的产品一致性
- C 阶段再推进更高层的平台装配能力

## 5. 暂不建议优先做的事项

在完成上述路线图前，不建议把以下事项放到更高优先级：

- 大规模新增未来模块页面
- 大规模扩展新量表但不先收口评估入口
- 扩展更多实验功能入口
- 在平台底座不稳的前提下继续堆叠“模块展示层”

## 6. 成功标准

当本路线图核心阶段完成后，SCGP 应达到的状态是：

- 当前主线数据可完整备份与恢复
- 资源中心的元数据和物理文件生命周期一致
- 评估中心入口和模块入口不再大量硬编码
- 未来模块以统一机制表达“开发中/未开放”
- 主产品入口和调试入口清晰分离
- 用户和文档看到的产品名称、模块状态、系统能力保持一致
