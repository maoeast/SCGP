# SCGP 当前产品需求文档（基于代码现状）

> **文档类型**: 现行 PRD
> **产品名称**: `SCGP / 星愿能力发展平台`
> **版本**: v2026-03-19
> **状态**: 当前主线基线
> **适用范围**: `main` 分支 2026-03-19 可见实现
> **编写依据**: `src/router/index.ts`、`src/main.ts`、`src/database/**`、`src/views/**`、`electron/main.mjs`
> **定位说明**: 本文描述当前代码已经具备的产品能力、明确可用的业务链路和已知边界；不把历史计划、实验能力或未来目标态写成当前已完成。

## 1. 产品概述

### 1.1 产品定位

SCGP（星愿能力发展平台）是一款面向特殊教育与康复场景的本地优先桌面平台，运行于 `Electron + Vue 3 + TypeScript + SQL.js` 技术栈之上。

当前平台已经从早期“生活自理适应综合训练系统”和“感官能力发展系统（SIC-ADS）”演进为一个具备平台化底座的综合应用，支持以下核心能力：

- 学生档案管理
- 标准化能力评估与报告
- 游戏训练
- 器材训练
- 训练计划与训练记录
- 统一资源中心
- 班级管理
- 本地账号、激活、备份与更新接入

### 1.2 当前产品边界

当前代码主线中，真正可运行、可通过主路由进入的业务能力主要集中在：

- 感官相关训练与评估
- 行为/发展类量表评估
- 训练计划编排与训练记录管理
- 资源中心与班级管理
- 系统级账号、激活、备份、更新、配置

平台层已经预留 `sensory`、`emotional`、`social`、`cognitive`、`life_skills` 等模块概念，但截至 **2026-03-13**：

- `sensory` 是当前最完整、最可用的业务模块集合
- `emotional`、`social` 已在部分计划、筛选项和模块菜单中出现
- 这些未来模块还不是完整可交付的独立业务模块

### 1.3 核心目标

当前产品阶段的核心目标不是再做单一量表工具，而是提供一套可持续演进的平台底座：

- 用统一学生档案承接评估、训练、计划和报告
- 用统一评估容器承接多量表接入
- 用统一资源模型承接游戏、器材、教学资料等素材
- 用统一记录与报告索引承接跨模块数据沉淀
- 用本地优先桌面架构满足离线、低部署成本和数据本地化需求

## 2. 用户与角色

### 2.1 当前角色

当前系统只有两类正式角色：

- `admin` 管理员
- `teacher` 教师

系统未实现更细粒度的功能点级权限模型，当前权限控制以角色级为主，并通过前端路由守卫和部分查询约束实现。

### 2.2 管理员职责

管理员当前可访问：

- 学生管理
- 能力评估与报告
- 游戏训练、器材训练、训练计划、训练记录
- 资源中心
- 系统管理
- 班级管理
- 学生分班
- 开发者调试与激活管理入口

管理员承担的典型职责：

- 维护系统账号
- 管理班级和学生分班
- 执行数据备份与恢复
- 配置系统参数
- 管理激活与版本更新
- 进入全部业务模块执行或审核操作

### 2.3 教师职责

教师当前可访问：

- 学生管理
- 能力评估与报告
- 游戏训练
- 器材训练
- 训练记录
- 训练计划
- 资源中心

教师承担的典型职责：

- 建立和维护学生档案
- 发起量表评估并查看报告
- 执行训练并记录结果
- 创建和执行训练计划
- 使用和浏览资源中心
- 查看自己负责班级相关数据

### 2.4 激活与登录约束

当前系统的访问门槛为：

1. 首次进入应用先检查激活状态
2. 若已正式激活，允许登录
3. 若未正式激活但仍处于试用期，允许继续登录和使用
4. 若未激活且试用期结束，只允许进入激活页

当前试用期为 **7 天**。

## 3. 当前信息架构

### 3.1 顶层模块

当前主路由中已经作为顶层入口存在的模块包括：

- 系统首页
- 学生管理
- 能力评估
- 游戏训练
- 训练记录
- 器材训练
- 报告生成
- 资源中心
- 训练计划
- 系统管理
- 班级管理
- 学生分班

### 3.2 模块化现状

系统内部已具备 `ModuleRegistry` 和模块元数据结构，但当前导航和路由仍主要是静态声明，不是注册表驱动的动态装配。

因此当前 PRD 以“真实路由可达模块”为准，而不以注册表中的未来模块声明为准。

### 3.3 首页现状

当前系统首页已经不再是简单的 ERP 式汇总页，而是以教师每日工作为中心的“特教业务指挥中心”。

当前首页已具备：

- 顶部核心指标卡片
- 快捷业务入口
- 今日训练日程
- 本周异常预警
- 待评估预警 / 智能特教助理提示

其数据口径当前明确为：

- 使用本地 SQLite / SQL.js 中的真实聚合查询
- 不再使用前端 Mock 数据填充首页业务列表
- 无数据时显示空状态，而不是伪造示例数据

当前首页还支持：

- 从今日训练日程中带 `studentId`、`planId`、资源上下文直接跳转到训练执行页
- 跳过原先重复的学生/模块选择步骤
- 对未授权模块入口进行灰态锁定和启动拦截

## 4. 功能需求

### 4.1 学生管理

当前能力：

- 学生列表展示
- 学生新增、编辑、删除
- 学生详情页
- 学生基础信息维护
- 学生头像/图片路径存储
- 学生相关评估与训练历史关联

业务价值：

- 统一学生主档案
- 让评估、计划、训练和报告围绕同一学生沉淀

### 4.2 能力评估

#### 4.2.1 当前已接入量表

当前评估中心可发起以下量表：

- `sm` 婴儿-初中生社会生活能力量表（S-M）
- `weefim` 改良儿童功能独立性评估量表（WeeFIM）
- `csirs` 儿童感觉统合能力发展评定量表
- `conners-psq` Conners 父母问卷
- `conners-trs` Conners 教师问卷
- `sdq` 长处和困难问卷
- `srs2` 社交反应量表第二版
- `cbcl` 儿童行为量表

#### 4.2.2 当前评估流程

统一主流程为：

1. 进入评估中心
2. 选择量表
3. 选择学生
4. 进入统一评估容器 `AssessmentContainer`
5. 完成评分与保存
6. 跳转对应报告页查看结果

#### 4.2.3 当前评估架构

当前评估链路已完成以下统一化：

- 使用统一评估容器承接多量表
- 使用 `ScaleDriver` 驱动量表题目、评分和报告逻辑
- 旧量表页面通过路由重定向进入统一容器
- 报告页按量表分别保留

当前仍未完全产品化的部分：

- 评估选择页仍是硬编码卡片，不是注册表动态生成
- 报告页仍是按量表分散实现
- 仍保留部分历史兼容和过渡结构

#### 4.2.4 当前评估结果管理

系统已支持：

- 单次评估保存
- 报告记录索引
- 多量表报告列表筛选
- 报告查看、下载、删除
- 部分历史报告数据迁移入口

### 4.3 游戏训练

#### 4.3.1 当前入口流程

当前流程为：

1. 进入游戏训练
2. 选择业务模块
3. 选择学生
4. 进入游戏大厅
5. 选择训练资源或具体游戏
6. 进入训练执行页
7. 保存训练记录与 IEP 报告相关结果

#### 4.3.2 当前能力范围

当前已实现的游戏训练能力包括：

- 基于学生上下文启动训练
- 训练过程中采集记录
- 训练结束后写入训练记录
- 支持 IEP 相关结果沉淀

当前代码中可见的游戏类型主要包括：

- 视觉网格类
- 视觉追踪类
- 音频类

#### 4.3.3 当前限制

- 当前真正可执行的业务主线仍以 `sensory` 为核心
- 菜单中出现的未来模块不是完整可交付训练链路

### 4.4 器材训练

#### 4.4.1 当前入口流程

当前流程为：

1. 进入器材训练
2. 选择模块

### 4.10 系统管理

#### 4.10.1 用户管理

当前系统支持：

- 管理员/教师账号管理
- 用户新增、编辑、删除
- 重置密码
- 登录日志记录与查询

#### 4.10.2 激活管理

当前系统支持：

- 激活页输入激活码
- 获取和显示机器码
- 试用期控制
- 激活状态校验
- 管理端激活相关入口

截至 2026-03-19，激活/授权能力已进一步收敛为：

- 激活码采用 RSA 验签
- 激活码 payload 必须包含 `am`（`allowedModules`）字段
- 当前实现不再为缺少 `am` 的旧格式激活码提供全量模块回退
- 本地 `activation` 表已单独持久化 `allowed_modules`
- 前端授权判断已统一收口为 `hasModuleAccess(moduleCode)`
- 路由、菜单、首页快捷入口与训练直达链路都已接入模块授权校验

当前开发环境还额外具备：

- `DEV` 环境下可跳过真实激活码验证
- 默认注入 `['sensory', 'emotional']` 作为开发期授权集合
- 该注入逻辑仅用于开发环境，不作为生产授权链路

#### 4.10.3 数据备份与恢复

当前系统支持：

- 导出加密备份文件
- 从备份文件恢复数据

但要明确：

- 当前备份方案是按表 JSON 导出，不是数据库文件级快照
- 当前备份覆盖范围落后于最新资源与班级扩展 schema

#### 4.10.4 系统设置

当前系统设置已支持：

- 系统名称

- Logo 等基础外观信息
- 自动备份选项
- 报告默认格式等参数

#### 4.10.5 软件更新

当前代码已接入更新服务，支持：

- 检查更新
- 下载更新
- 退出并安装更新
- 跳过版本
- 自动更新开关
- 下载进度与错误提示

需要注意：

- 更新依赖 Electron 更新处理器是否可加载
- 若相关模块缺失，应用仍能启动，但自动更新不可用

## 5. 核心业务流程

### 5.1 激活与登录流程

1. 应用启动
2. 初始化数据库、模块注册和策略
3. 恢复登录状态
4. 检查激活状态
5. 在生产环境中，若未激活且试用结束，进入激活页
6. 在生产环境中，若已激活或在试用期内，允许登录
7. 登录后由模块授权状态决定是否允许进入对应业务模块
8. 在开发环境中，可使用内置 DEV 授权注入继续测试主线模块

### 5.2 评估流程

1. 教师选择量表
2. 教师选择学生
3. 进入统一评估容器
4. 系统按量表驱动加载题目和评分逻辑
5. 提交后保存评估结果
6. 生成并进入报告页
7. 报告被索引到统一报告列表

### 5.3 训练流程

1. 教师从游戏训练、器材训练或训练计划发起训练
2. 选择学生和资源
3. 完成训练执行
4. 系统写入训练记录
5. 必要时生成 IEP 相关报告或建议
6. 训练结果进入训练记录与报告索引体系

当前还补充支持：

- 教师可从首页“今日训练日程”直接发起训练
- 若计划已绑定可启动资源，则系统可带着学生与计划上下文直接进入训练执行页
- 若目标模块未授权，则启动前会被拦截

### 5.4 资源管理流程

1. 管理员进入资源中心
2. 按模块、类型、分类或关键词查看资源
3. 管理资源状态和元数据
4. 资源通过 `resource://` 协议供前端展示
5. 资源被训练、计划和其他业务链路复用

## 6. 数据与架构要求

### 6.1 当前技术栈

- 桌面壳：Electron
- 前端：Vue 3 + TypeScript + Element Plus + Vue Router + Pinia
- 数据库：SQL.js
- 打包与开发：Vite

### 6.2 当前持久化模型

当前主数据持久化链路为：

1. 渲染进程内运行 SQL.js
2. 数据变更后通过 `SQLWrapper` 防抖保存
3. 导出数据库二进制
4. 通过 IPC 发送到 Electron 主进程
5. 主进程使用 `.tmp + fsync + rename` 原子写入到 `userData/database.sqlite`

### 6.3 当前文件与资源模型

- 用户资源目录：`userData/resources/`
- 预置资源目录：`assets/resources/` 或打包后的 `process.resourcesPath/assets/resources/`
- 资源展示协议：`resource://`

### 6.4 当前模块配置模型

`ModuleRegistry` 已存在，模块配置当前主要通过 `localStorage` 保存，而不是完全由数据库统一托管。

### 6.5 当前安全模型

当前安全策略主要包括：

- 角色级访问控制
- 激活码 RSA 验签
- 本地机器码绑定
- 基于 `allowedModules` / `hasModuleAccess` 的模块授权拦截
- 资源路径校验与目录遍历防护

当前不应被误写为已完全具备的能力：

- 强硬件级机器码
- 服务端统一鉴权
- 细粒度权限中心

## 7. 非功能需求

### 7.1 运行要求

- 本地优先
- 支持离线运行主要业务链
- 不依赖在线后端作为主链
- Electron 环境和浏览器调试环境都可运行前端

### 7.2 数据完整性

- 关键数据写入必须可落盘
- 应用退出前要触发立即保存
- 数据库文件写入使用原子替换策略

### 7.3 可维护性

- 新量表接入应优先复用统一评估容器和 `ScaleDriver`
- 新资源应优先进入统一资源模型
- 新模块扩展应优先延续平台底座而非再造垂直孤岛

### 7.4 兼容性

- 保留旧评估路由到统一容器的重定向
- 保留部分历史数据迁移能力
- 保留旧名称与旧结构的有限兼容，但不再作为当前命名标准

## 8. 当前已知限制与产品债

以下内容应写入当前产品现实，而不是被忽略：

- 模块注册已存在，但导航与路由仍是静态装配
- `sensory` 是当前最完整业务主链，其余未来模块仍偏占位
- 资源收藏筛选未完成
- 备份/恢复未完全覆盖新资源表与班级扩展表
- 资源物理文件生命周期管理仍不够统一
- 激活机器码可靠性有限
- `Reports`、`ResourceCenter`、`TrainingPlan` 等跨模块页面的授权语义仍需进一步细化
- 登录 token 与密码哈希方案偏轻量
- 系统标题、缓存 key、部分元数据仍残留旧产品名
- 主路由树中仍保留较多开发和迁移工具入口

## 9. 当前版本结论

截至 **2026-03-19**，SCGP 应被定义为：

“一个已经具备平台底座雏形、并已在学生档案、标准化评估、感官训练、器材训练、训练计划、训练记录、资源中心、班级管理和本地系统能力上形成可运行主链的特殊教育桌面平台。”

它不是：

- 仅仅一个生活自理量表系统
- 已经完成所有未来模块的平台成品
- 完全动态模块化、完全收敛技术债的最终态架构

后续文档如果描述当前产品，应优先以本文、`README.md`、`重构实施技术规范.md`、`PROJECT_CONTEXT.md` 和最新专项报告为准。
