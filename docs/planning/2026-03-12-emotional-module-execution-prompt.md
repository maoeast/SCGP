# 情绪行为调节模块执行版提示词

## 目标

请基于当前 SCGP 项目的真实代码结构，为系统新增“情绪行为调节模块（Emotional Module）”，并输出一份可直接指导后续设计与开发的实施方案。

## 项目真实上下文

1. 当前项目是一个已经存在的本地优先桌面系统，不是从零开始的新项目。
2. 技术栈固定为：
   - Electron
   - Vue 3
   - TypeScript
   - Vite
   - SQL.js
   - ECharts
3. 当前项目中已经存在：
   - 模块注册机制
   - `ModuleCode.EMOTIONAL`
   - 游戏训练模块入口
   - 学生选择流程
   - 训练记录模块
   - 报告列表模块
   - 本地资源存储与统一资源查询能力
4. 当前 `emotional` 模块在代码中已被预留为 `experimental` 状态，因此本次目标是将其设计为可正式接入现有系统的训练模块。
5. 系统为纯前端本地离线应用，不允许依赖后端服务。

## 业务目标

本次新增模块的核心目标是帮助特殊儿童：

1. 识别基础情绪。
2. 理解情绪背后的生活场景和社交线索。
3. 学会恰当表达关心。
4. 通过可重复练习提升社交推理和回应能力。
5. 让老师和家长看到训练数据、训练偏好和进步趋势。

## 目标用户与场景

目标用户：

- 特殊学校学生
- 孤独症谱系、发育迟缓、社交理解困难儿童
- 教师
- 家长

使用场景：

- 特殊学校课堂教学
- 个训与小组训练
- 家庭康复训练
- 教师/家长复盘学生表现

## 功能范围

本次首期完整实现两个子模块。

### 子模块 1：情绪与场景

1. 场景识别
   - 系统数据结构需支持不少于 80 个真实生活场景资源。
   - 首期实现允许只上线少量演示数据，但必须保证后续能扩展到 80+。
   - 场景示例包括：生日派对、医院探望、学校课堂、超市购物、摔倒受伤、等待排队、被拒绝、被表扬、丢失物品、同伴冲突等。
   - 支持点击切换场景。
2. 情绪解读
   - 针对每个场景，引导用户从“开心、失落、尴尬、生气、害怕”5 种基础情绪中选择。
   - 选择后立即给出反馈。
   - 反馈需区分：
     - 正确反馈
     - 错误反馈
     - 轻提示后重试
     - 多次错误后的示范讲解
3. 社交推理
   - 情绪识别正确后进入 2-3 个引导性问题。
   - 问题必须和场景强绑定。
   - 问题类型至少包括：
     - 原因推理
     - 需求推理
     - 自我代入
   - 每题需要定义：
     - 标准答案
     - 可接受答案
     - 错误反馈
     - 提示策略
4. 问题解决
   - 每个场景提供 3-4 种应对方案。
   - 用户选择后，需要解释“为什么这个方案更好”。
   - 允许存在：
     - 最佳方案
     - 可接受方案
     - 不推荐方案

### 子模块 2：表达关心

1. 情境练习
   - 系统数据结构需支持不少于 60 个“需要表达关心”的场景资源。
   - 首期允许少量演示内容上线，但后续必须可扩展到 60+。
   - 场景示例包括：朋友考试失利、家人身体不适、同学摔倒、同伴哭泣、被批评、一个人沉默等。
2. 话术训练
   - 每个场景提供 3 种关心表达方式：
     - 共情式
     - 建议式
     - 行动式
   - 选择后展示该话术的使用效果说明。
   - 需要解释：
     - 为什么这句话更让人舒服
     - 为什么某句话不够合适
3. 视角切换
   - 支持“表达者视角”和“接收者视角”。
   - 表达者视角：
     - 选择话术
     - 查看接收者虚拟反应
     - 反馈形式为文字描述 + 简单表情
   - 接收者视角：
     - 展示不同话术
     - 让用户选择“哪句话最舒服”
     - 说明原因
4. 进度追踪与报告
   - 自动记录：
     - 场景 ID
     - 选择结果
     - 是否正确
     - 提示次数
     - 重试次数
     - 作答时间
     - 关心话术偏好
     - 视角切换结果
   - 生成情绪模块专属可视化报告。
   - 至少包含：
     - 情绪识别正确率趋势图
     - 场景类型掌握分布图
     - 关心话术类型偏好图
     - 回答时长趋势图
     - 表达者/接收者视角表现对比图

## 特殊教育交互要求

1. 单屏单任务，减少干扰。
2. 按钮大、文本短、口语化、适合触屏。
3. 图文双编码，必要时支持语音提示。
4. 正向反馈优先，错误反馈要温和。
5. 阶梯式提示：
   - 第一次错误：轻提示
   - 第二次错误：视觉线索或缩小选项
   - 第三次错误：给出示范并解释原因
6. 训练节奏可控，支持中断续做。
7. 支持教师辅助模式。
8. 不使用高刺激动画和复杂页面。
9. 支持能力分层：
   - 初级：识别情绪
   - 中级：理解原因和需求
   - 高级：选择回应并说明原因

## 专业逻辑要求

1. 所有情绪分类、社交推理和关心话术必须符合特殊教育“阶梯式干预”原则。
2. 可参考《自闭症儿童社交情绪训练指南》的训练思路，但内容表达必须适合儿童理解。
3. 每个场景内容必须结构化，避免写成零散文案。
4. 需要考虑同一情绪跨多个场景的泛化训练，以及同一场景下不同表达方式的对比训练。

## 技术设计要求

1. 必须优先复用现有架构：
   - 模块注册
   - 模块菜单入口
   - 学生选择流程
   - 训练记录体系
   - 报告体系
   - 资源存储体系
2. 资源内容建议基于现有资源表扩展，使用 `module_code + resource_type + metadata` 承载场景、问题、话术、方案等内容。
3. 行为记录必须兼容现有本地数据库。
4. 对训练数据存储，请对比两种方案并给出推荐：
   - 方案 A：继续复用现有训练记录表，在 `raw_data` 中存储情绪模块明细
   - 方案 B：新增情绪模块专用明细表
5. 报告图表必须使用 ECharts。
6. 方案必须支持纯本地离线查看、统计与导出。
