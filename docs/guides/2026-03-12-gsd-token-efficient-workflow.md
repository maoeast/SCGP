# SCGP 项目下 GSD 的低 Token 开发工作流

## 1. 文档目的

本文档用于约束 SCGP 项目在使用 GSD 工作流时的默认策略，目标不是追求“流程最完整”，而是追求“在保证质量的前提下尽量降低 token 消耗、减少长对话上下文堆积、提高恢复效率”。

适用场景：

- 中大型功能持续开发
- 跨天、多轮协作
- 需要在 Codex/GSD 工作流下推进任务
- 希望减少超长会话造成的 token 累积

## 2. 核心结论

对 SCGP 这种大仓库，最省 token 的方式不是把所有事情都留在聊天线程里，而是：

1. 把长期上下文沉淀到文档。
2. 把阶段状态沉淀到 GSD 状态文件。
3. 大功能只做一次完整规划。
4. 日常开发尽量用轻量路径。
5. 阶段收尾时再做验证，而不是每一小步都做全套流程。

## 3. 默认工作原则

### 3.1 大功能用 Phase，小任务用 Quick

默认规则：

1. 新模块、新架构、跨模块改造，用 `gsd-plan-phase`。
2. 页面小改、路由补充、表字段增补、局部修复，用 `gsd-quick`。

判断标准：

- 如果任务需要重新定义边界、数据结构、页面关系、验收标准，属于 Phase。
- 如果任务已经知道怎么做，只差动手实现，属于 Quick。

### 3.2 不用聊天历史当状态存储

不要依赖“前面聊过什么”来恢复工作。

应使用：

- `gsd-progress`
- `gsd-pause-work`
- `gsd-resume-work`
- `.planning/` 中的阶段文档

这样做的好处：

1. 新会话不用重新喂很长历史。
2. 恢复上下文时只读关键文件。
3. 避免一天内反复读取旧线程导致缓存读取量过大。

### 3.3 需求尽量写入 `.md`，不要反复口述

推荐做法：

1. 需求先落到 `docs/planning/*.md`
2. 页面设计落到 `docs/architecture/*.md`
3. 实施计划落到 `docs/plans/*.md`

后续所有 GSD 执行尽量引用文档路径，而不是在对话里重复粘贴整段需求。

### 3.4 验证只在阶段完成时做

不要每改一个文件就重新跑完整的计划校验、执行校验、阶段校验。

推荐：

1. 规划时校验一次
2. 阶段收尾时校验一次
3. 日常局部实现只做必要的类型检查和手工验证

## 4. SCGP 推荐命令分工

## 4.1 `gsd-progress`

使用时机：

- 每天开始工作时
- 切会话后恢复上下文时
- 不确定当前项目推进到哪一步时

作用：

- 查看当前阶段
- 查看推荐下一步动作
- 避免先在聊天里大量追问历史状态

推荐频率：

- 每次开工前 1 次即可

## 4.2 `gsd-plan-phase`

使用时机：

- 新模块建设
- 跨模块重构
- 需求边界复杂
- 需要形成正式 `PLAN.md`

SCGP 中适用任务示例：

- 新增情绪行为调节模块
- 重构训练记录模型
- 改造报告体系
- 新评估模块接入统一容器

省 token 用法：

1. 优先使用 `--prd <file>`
2. 如果需求已明确，优先 `--skip-research`
3. 不要对同一个 phase 反复重新规划

推荐示例：

```bash
$gsd-plan-phase 05 --prd docs/planning/2026-03-12-emotional-module-prd.md --skip-research
```

## 4.3 `gsd-quick`

这是 SCGP 日常开发中最应该高频使用的命令。

使用时机：

- 需求已明确
- 改动范围有限
- 不需要完整 phase ceremony

SCGP 中适用任务示例：

- 增加一个路由
- 新增一个页面骨架
- 补一个 migration
- 修复一个训练记录筛选 bug
- 补一个报告图表

推荐模式：

1. 默认直接 `gsd-quick`
2. 有轻微歧义时 `gsd-quick --discuss`
3. 风险稍高但还不值得全 phase 时 `gsd-quick --full`

示例：

```bash
$gsd-quick 新增 emotional 模块 Report.vue 页面骨架并接入路由
```

```bash
$gsd-quick --full 为情绪模块新增训练记录明细表和查询 API
```

## 4.4 `gsd-pause-work`

使用时机：

- 当天做不完
- 需要切到别的任务
- 当前任务已进入中间状态

作用：

- 把当前上下文沉淀下来
- 下次恢复时减少重新解释背景的成本

推荐频率：

- 每次中断一个中大型任务时都应执行

## 4.5 `gsd-resume-work`

使用时机：

- 隔天继续
- 切会话后继续
- 希望快速恢复而不是翻长线程

作用：

- 用文件恢复上下文，替代长对话回放

## 4.6 `gsd-add-todo`

使用时机：

- 讨论过程中冒出旁支需求
- 当前不准备立即实现

作用：

- 先记下来，防止主任务线程越聊越大

SCGP 中典型例子：

- “后续支持教师自定义情绪场景”
- “二期支持语音播报”
- “后续补报告导出”

## 4.7 `gsd-validate-phase`

使用时机：

- 一个 phase 基本完成后
- 需要查漏补缺

作用：

- 做阶段级验证
- 补齐 Nyquist 验证缺口

不推荐用法：

- 每完成一个小功能就跑一次

## 5. SCGP 下最省 Token 的标准工作流

### 5.1 新功能启动

步骤：

1. `gsd-progress`
2. 将需求写入 `docs/planning/*.md`
3. 如需补页面和数据设计，写入 `docs/architecture/*.md`
4. 只进行一次 `gsd-plan-phase`

推荐模式：

```text
gsd-progress
-> 整理 PRD / 数据设计 / 页面设计
-> gsd-plan-phase <phase> --prd <file> --skip-research
```

### 5.2 日常实现

步骤：

1. 从计划中选一个原子任务
2. 使用 `gsd-quick`
3. 完成后更新状态
4. 若未完成则 `gsd-pause-work`

推荐模式：

```text
读 PLAN.md
-> gsd-quick 实现单个子任务
-> 类型检查 / 手工验证
-> 继续下一项或 pause
```

### 5.3 阶段收尾

步骤：

1. 汇总已完成项
2. 运行必要验证
3. 使用 `gsd-validate-phase`
4. 补文档和总结

## 6. 最容易浪费 Token 的错误用法

1. 一个简单任务也走完整 `plan -> execute -> verify`。
2. 每轮都重新口述一遍需求、上下文和历史。
3. 已经写进文档的内容继续在聊天里长篇重复。
4. 一个小改动开过多 agent。
5. 同一个 phase 反复重新规划。
6. 不做 pause / resume，长期拖着一个超长线程。

## 7. 推荐的会话粒度

对 SCGP 这种项目，推荐会话粒度如下：

1. 一个会话只推进一个明确目标。
2. 一个会话最好只覆盖一个 phase 的一个子任务，或几个高度相关的小任务。
3. 做完一批任务后结束会话。
4. 下一次从 `gsd-progress` 或 `gsd-resume-work` 恢复。

这样通常比单线程长聊一整天更省 token。

## 8. 建议的命令优先级

默认优先级：

1. `gsd-progress`
2. `gsd-quick`
3. `gsd-pause-work` / `gsd-resume-work`
4. `gsd-plan-phase`
5. `gsd-validate-phase`

含义：

- `gsd-quick` 应该是主力，不是例外
- `gsd-plan-phase` 应该用于大节点，而不是日常所有任务

## 9. 面向情绪模块的使用建议

如果后续继续推进情绪行为调节模块，推荐：

1. 用现有 PRD 和设计文档作为唯一需求输入源。
2. 只做一次 phase 规划。
3. 后续每个子任务都走 `gsd-quick`。
4. 完成“入口、数据、训练页、记录页、报告页”后再做 phase 验证。

## 10. 最终建议

SCGP 项目下，想省 token，就遵守下面这条默认策略：

1. 大功能先写文档，再做一次 Phase 规划。
2. 实现阶段默认 `gsd-quick`。
3. 断点必须 `pause`，恢复必须 `resume` 或 `progress`。
4. 不用长对话线程承载项目状态。
