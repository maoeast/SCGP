# 2026-03-13 SCGP PRD 差距清单

> **文档类型**: 差距分析
> **基线文档**: `docs/planning/2026-03-13-scgp-current-prd.md`
> **分析对象**: 2026-03-13 当前代码主线
> **目的**: 明确“现行 PRD 已定义的产品方向”与“当前代码仍处于过渡态或未完成的部分”

## 1. 结论摘要

当前 SCGP 已经具备平台化底座和可运行主链，但离“平台化产品形态稳定”仍有明显差距。

最核心的判断是：

- 当前可交付主链已经成立
- 当前平台抽象已经存在
- 但平台抽象尚未完全转化为产品层的统一行为

换句话说，当前系统更像“已跑通主链的平台化过渡版本”，而不是“完全收敛的多模块平台正式版”。

## 2. 高优先级差距

### 2.1 模块化架构已存在，但导航和装配仍是静态的

现状：

- `ModuleRegistry` 已存在
- 模块元数据、特性列表、配置结构已经定义
- 但实际路由、页面入口和菜单仍主要在 `src/router/index.ts` 中静态声明

影响：

- 新模块接入成本仍偏高
- 模块注册表无法真正成为产品层单一事实来源
- 文档容易再次把“注册表目标态”误写为“当前已动态装配”

建议：

- 下一步把“模块菜单 + 路由入口 + 模块状态展示”逐步改为注册表驱动
- 至少先做到模块首页和模块卡片由注册表生成

### 2.2 未来模块已出现在 UI 中，但尚未形成完整可交付链路

现状：

- `emotional`、`social` 已出现在模块菜单、计划筛选等位置
- 但当前真正完整可运行的业务主链仍以 `sensory` 为主
- 这些未来模块更多是占位、规划入口或局部能力暴露

影响：

- 用户会误以为平台已具备完整多模块交付能力
- PRD 与真实交付边界容易再次漂移

建议：

- 在界面上明确标识“实验中 / 开发中 / 未开放”
- PRD、路标和系统内文案统一写清“当前可用模块”和“预留模块”

### 2.3 备份/恢复方案未覆盖当前主线 schema

现状：

- 当前备份仍偏向旧表导出
- `sys_training_resource`、`sys_tags`、`sys_resource_tag_map`
- 班级相关扩展表与部分新主线表未被完整纳入当前备份逻辑

影响：

- 用户认为“已备份全部系统数据”，但实际恢复结果可能不完整
- 这是实质性数据风险，不只是文档问题

建议：

- 先补齐当前 schema 覆盖范围
- 再决定是否升级到数据库文件级快照 + 资源目录打包

### 2.4 资源文件生命周期管理尚未完全收敛

现状：

- 资源元数据模型已经统一到 `sys_*`
- `resource://` 协议已经可用
- 但资源物理文件保存、删除、清理仍缺少更统一的主链约束

影响：

- 资源元数据与物理文件可能出现漂移
- 自定义资源长期运营时容易出现孤儿文件或引用失效

建议：

- 明确统一的资源文件服务层
- 把新增、替换、删除、硬删除、恢复的文件操作收口到同一主链

## 3. 中优先级差距

### 3.1 资源收藏能力未真正完成

现状：

- 资源查询接口里 `favoritesOnly` 仍是 `TODO`
- 收藏相关结构没有形成完整用户体验

影响：

- 资源中心缺失一个高频使用能力
- “统一资源中心”在用户体验层仍不完整

建议：

- 先完成收藏查询
- 再补收藏入口、统计和筛选交互

### 3.2 评估中心仍存在“统一执行、分散入口”的过渡形态

现状：

- 评估执行层已经统一到 `AssessmentContainer + ScaleDriver`
- 但评估选择页仍是硬编码卡片
- 报告页也仍是按量表分散实现

影响：

- 接入新量表时仍需要改多个层面
- 平台能力和 UI 形态没有完全一致

建议：

- 先将评估选择页改为配置驱动
- 报告页再逐步抽公共骨架

### 3.3 系统命名清理尚未完成

现状：

- 文档主入口已统一到 `SCGP`
- 但代码中仍残留旧名称默认值、缓存 key、备份元数据等

影响：

- 用户界面和文档口径仍可能偶发不一致
- 不利于后续品牌统一和交付一致性

建议：

- 统一清理默认系统名、缓存 key 命名、备份 metadata 命名
- 同时保留必要的旧值兼容迁移

### 3.4 开发/迁移工具仍混在主路由树中

现状：

- 路由树中仍保留大量 devtools、迁移验证、测试入口

影响：

- 生产产品边界不够干净
- 后续维护者容易把工具页误认为正式产品能力

建议：

- 明确将调试工具收敛到开发模式专用入口
- 或进一步隐藏到单独的调试开关下

## 4. 低优先级但需要记录的差距

### 4.1 认证和本地安全策略偏轻量

现状：

- 登录 token 为简单本地编码
- 本地密码哈希策略较轻量
- 更适合单机离线桌面场景，不适合被误描述为高强度安全方案

建议：

- PRD 和外部材料中不要夸大安全等级
- 如果未来进入更高安全场景，再单独升级本地身份和密钥管理方案

### 4.2 机器码可靠性有限

现状：

- 当前机器码方案更多是可用性优先，不是强硬件绑定

影响：

- 激活体系适合当前离线桌面授权模型
- 但不宜被当作强对抗授权方案

建议：

- 在产品和运营层面合理设定预期
- 未来如需要更强机器绑定，再升级主进程采集策略

## 5. 建议的执行顺序

建议后续按以下顺序推进：

1. 先补齐备份/恢复对当前 schema 的覆盖
2. 再收敛资源文件生命周期管理
3. 再完成资源收藏与资源中心体验收口
4. 再推进评估中心入口配置化
5. 再推进模块首页/导航注册表驱动
6. 最后清理命名残留和开发工具暴露面

## 6. 适合作为下一阶段目标的事项

如果要从当前 PRD 往下拆执行阶段，最适合单列成阶段目标的有三块：

- Phase A: 备份、资源文件、收藏等平台底座收口
- Phase B: 评估中心和资源中心的配置化/平台化
- Phase C: 模块注册表驱动的导航与模块开放机制

## 7. 结论

当前 SCGP 的问题已经不再是“有没有主链”，而是“主链之上的平台行为是否已经统一收口”。

因此接下来的重点不应再回到旧产品式的大而泛功能堆叠，而应优先处理以下三类平台债：

- 数据可靠性债
- 资源生命周期债
- 模块装配一致性债

只要这三块逐步收口，SCGP 才会从“平台化过渡版本”真正进入“平台化产品版本”。
